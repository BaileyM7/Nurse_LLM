1. System Design & Implementation Plan
Rubric criteria: Restate use case/constraints, present system architecture, justify stack choices, define success criteria.
1.1 Use Case Restatement
Nursing students use an AI-powered chatbot to practice patient assessment through simulated patient interactions and receive structured feedback on assessment completeness. The system targets nursing education environments (simulation labs, coursework, self-study).
Constraints (Unchanged from P1)
* Privacy/Compliance: No live patient data; all scenarios use public or synthetic cases. FERPA considerations for student interaction logs.
* Cost: OpenAI API token usage must remain within academic budget (~$50/month). Context window management is critical.
* Reliability: System must gracefully handle API failures and not produce clinically dangerous hallucinations.
* Scope: Limited to assessment practice only — no diagnosis, treatment, or prescription guidance.
1.2 System Architecture
The system follows a three-layer architecture. Below is a detailed description of each component and how data flows through the pipeline.
Layer
	Components
	Details
	Data Layer
	Case Database + Scenario Schema
	30–50 structured patient cases from OpenLabyrinth, PhysioNet (discharge summaries), and synthetic scenarios. Each case encoded as JSON with: demographics, chief complaint, history, symptoms (present/absent), expected assessment questions, and rubric.
	Application Layer
	LangChain Orchestration + OpenAI API + Assessment Tracker
	LangChain manages prompt templates, conversation memory, and chain-of-thought routing. The Assessment Tracker monitors which clinical domains the student has explored (e.g., HPI, ROS, PMH, medications, allergies, social history) and updates coverage in real time.
	Presentation Layer
	Streamlit Web UI
	Chat interface with: scenario selector, live conversation window, assessment coverage dashboard (progress bar per domain), and post-session feedback report. Deployed locally or on Streamlit Cloud for user testing.
	

Data Flow
1. User selects a scenario by patient’s name (BOB)  from the case library via the Streamlit UI.
2. The scenario JSON is loaded and injected into the system prompt as the patient persona.
3. User types assessment questions; LangChain routes them through the conversation chain.
4. OpenAI API (GPT-4o-mini) generates patient responses, constrained by the scenario data (only reveals symptoms listed in the case).
5. The Assessment Tracker parses each user question, classifies it into a clinical domain, and updates the coverage matrix.
6. When the uses ends the session (or hits a turn limit), the Feedback Generator produces a structured report: domains covered vs. missed, quality of questioning, and suggestions.
1.3 Stack Justification
Tool
	Choice
	Justification
	LLM
	GPT-4o-mini (OpenAI)
	Best cost/quality ratio for conversational simulation. Supports structured output and function calling for assessment tracking. Fallback: GPT-3.5-turbo if budget constrained.
	Orchestration
	LangChain (Python)
	Mature prompt templating, conversation memory (buffer + summary), and chain composition. Enables swapping LLM providers without rewriting logic.
	UI
	Streamlit
	Rapid prototyping of chat interfaces with built-in session state. Sufficient for academic prototype; no frontend engineering overhead.
	Data Storage
	JSON files + SQLite
	JSON for scenario definitions (version-controlled). SQLite for session logs (student interactions, timestamps, coverage data) to support evaluation.
	Deployment
	Streamlit Cloud / Local
	Free tier for academic use. Local deployment for user study sessions to control latency and data.
	Evaluation
	Python (scikit-learn, pandas)
	Standard libraries for computing metrics, generating plots, and running statistical tests on feedback quality.

3. Testing & Evaluation Readiness
Rubric criteria: Specify testing plan, include baseline comparison, address deployment concerns.
3.1 Evaluation Plan
Evaluation Type
	Method
	Metrics
	Offline — Response Fidelity
	Manual annotation of 100 sampled conversation turns (2 annotators). Check: Does the patient response stay faithful to the scenario? Any hallucinated symptoms?
	Fidelity rate (%), inter-rater agreement (Cohen’s kappa)
	Offline — Domain Classification
	Held-out test set (20% of annotated questions). Each student question labeled with ground-truth assessment domain.
	Accuracy, precision/recall per domain, confusion matrix
	Offline — Feedback Quality
	Compare system-generated feedback reports against expert rubric for 20 complete sessions. Faculty member rates agreement.
	Agreement rate (%), qualitative gap analysis
	Functional — Edge Cases
	Test with adversarial inputs: off-topic questions, requests for diagnosis/treatment, gibberish, attempts to break character.
	Pass/fail per edge case category, guardrail trigger rate
	User Study
	8–12 nursing students complete 2–3 scenarios each. Post-session survey (Likert scale) + open-ended feedback. Compare perceived learning vs. expectations.
	Satisfaction score, engagement metrics (turns, time, completion), qualitative themes
	

3.2 Baselines for Comparison
* Simple baseline (rule-based scripted patient): Fixed keyword-response mappings. No conversational flexibility, no tracking. Represents the floor.
* Relevant baseline (few-shot LLM without tracking): GPT-4o-mini with scenario context but no assessment tracker or feedback generator. Isolates the value of the tracking/feedback components.
* Optional stretch baseline: Existing commercial nursing simulation tool (if accessible) or published benchmarks from nursing education literature.
3.3 Failure Modes & Diagnostic Checks
Failure Mode
	Detection Method
	Mitigation
	Hallucinated symptoms
	Compare patient responses against scenario symptom_map. Flag any symptom mentioned that isn’t in the case definition.
	Add explicit negative constraint in system prompt: “Only mention symptoms listed in the case. If asked about unlisted symptoms, deny them.”
	Character breaks
	Detect when the LLM drops the patient persona (e.g., says “As an AI” or gives medical advice).
	Strengthen persona instructions. Add post-processing filter to catch and regenerate broken responses.
	Domain misclassification
	Confusion matrix analysis on held-out test set. Identify which domain pairs are most confused.
	Add few-shot examples for ambiguous question types. Consider multi-label classification for questions spanning domains.
	Feedback too generic
	Manual review: does the feedback reference specific student actions or just repeat template text?
	Switch from template-based to LLM-generated feedback with explicit instructions to cite specific conversation turns.
	API rate limits / failures
	Monitor API response codes and latency.
	Implement retry logic with exponential backoff. Cache common scenario prompts. Fallback to GPT-3.5-turbo.

	Severity
	Mitigation
	Decision Point / Fallback

	Medium
	Augment with 10–15 synthetic cases co-designed with a nursing faculty advisor. Validate realism with 2–3 nursing students before full study.
	If cases still feel unrealistic by Week 6 → reduce scope to 15–20 high-quality cases and focus evaluation on those.

	Medium
	Use GPT-4o-mini (cheaper). Limit conversation to 30 turns. Summarize conversation history after 15 turns to reduce context.

	Assessment tracker accuracy too low
	High
	Iterative prompt engineering with error analysis. Add few-shot examples for each domain. Consider fine-tuning a small classifier as fallback.
	If accuracy stays <75% by Week 6 → simplify to binary tracking (asked/not asked per domain) instead of nuanced classification.

	If <5 participants by Week 7 → pivot to expert evaluation only (3 nursing faculty rate the system).
	Scope creep into diagnosis
	Low
	Hard guardrails in system prompt. Post-processing filter rejects responses containing diagnostic language.
	Already mitigated by design. Monitor in testing.
	Integration complexity
	Medium
	Modular architecture: each component testable independently. Integration tests run after each sprint.
	If end-to-end integration fails → present strong offline prototype + evaluation + screenshots as deliverable.
